# Sportsbook Platform Deep Dive

Last reviewed: 2026-04-20

## Summary

This repo is a working private NCAAB-first betting research platform with early
multi-sport scaffolding. ESPN schedule/results collection, event/team/league
schema, API sport filters, and several Streamlit selectors support NCAAB, NCAAF,
NFL, and MLB through the sport/provider registry. NBA and soccer are planned
registry placeholders and are not active collection or UI sports.

The main product flow is the Sportsbook Board backed by `GET /board` and
`GET /events/{event_id}/research`. It has cached board/research calls, expandable
game rows, line buttons, and a session-state Research Slip. It is the closest
thing to a daily-use betting workflow, but populated-state desktop/mobile polish
is still unverified.

The production foundation is now documented as Debian VM + SQLite + Tailscale +
localhost systemd services + cron. Docker Compose, APScheduler, Windows Task
Scheduler, and older cloud deploy scripts still exist and should be treated as
legacy or alternative paths unless a future decision replaces this profile.

## Important Files

- `dk_ncaab/collectors/load_games.py`: ESPN schedule/results loader for NCAAB,
  NCAAF, NFL, MLB.
- `dk_ncaab/config/sports.py`: source of truth for sport/provider eligibility,
  labels, provider keys, enrichers, and UI availability.
- `dk_ncaab/collectors/odds_api.py`: DraftKings odds via The Odds API for the
  same four sports, but current default odds config enables only `baseball_mlb`.
- `dk_ncaab/collectors/splits_dknetwork.py`: Action Network public splits;
  currently NCAAB-only and Playwright-heavy.
- `dk_ncaab/db/models.py`: league/team/event schema is multi-sport-capable;
  there is no player schema yet.
- `dk_ncaab/etl/features.py`: feature rows are generic by event/market/side but
  still call NCAAB-specific KenPom/AP enrichers.
- `api/main.py`: read-only API has 17 routes including docs routes, board,
  research, games, teams, standings, model, backtest, status, and runs.
- `ui/pages/sportsbook_board.py`: healthiest UI flow; sport choices come from
  the registry.
- `scripts/cron_collect_cycle.sh`: current safe one-shot cron wrapper; odds and
  splits disabled unless env flags opt in.
- `scripts/install_systemd_services.sh`: localhost API/UI services and optional
  Tailscale Serve publishing.

## Key Findings

- Free ESPN schedule/results is the only truly multi-sport collection path today.
- The Odds API fan-out loops sports, but durable quota accounting is incomplete
  and multi-sport calls can be undercounted by orchestration paths. Later on
  2026-04-20 this was addressed with append-only `odds_api_usage` rows and
  pre-HTTP budget/cadence gates in `collect_odds()`.
- SQLite is the production default now. Postgres Docker compose remains for
  local/legacy experiments; keep code migration-friendly.
- App privacy is mostly perimeter-based. The current systemd path binds services
  to localhost and uses Tailscale; CORS is restricted by config, but there is
  still no app-level auth.
- Modeling/backtests are exploratory. Row-level temporal CV can leak across
  rows from the same event, close-aware features are used in outcome models, and
  EV is not settlement-accurate yet.
- Player/team/signal expansion is thin. There is no player table, injury feed,
  prop schema, player stat provider, saved edge history, or persistent bet-slip
  style review workflow.
- UI performance risks come from sequential DB/API work: board/research, games,
  standings, and team history repeatedly fetch quote/snapshot/team details.
- `artifacts/models` is absent in this checkout; parquet feature artifacts exist
  and `features_trainable.parquet` was verified at 342 rows by 89 columns.

## Recommended Next Moves

1. Keep SQLite-on-VM as the production storage default unless a future decision changes it.
2. Make cron/systemd/Tailscale the single production path, or clearly quarantine
   Docker/APScheduler/Windows paths as legacy.
3. Harden ESPN schedule/results tests for NCAAB, NCAAF, NFL, and MLB before
   enabling NBA/soccer.
4. Add event-grouped walk-forward validation, out-of-fold predictions, and
   settlement-accurate EV/ROI math before promoting signals as actionable.
5. Design player/injury/prop schemas only after choosing data providers for NBA,
   NFL, MLB, NCAAF, NCAAB, and soccer.
6. Batch hot API queries and add API contract tests plus populated desktop/mobile
   screenshots for the Sportsbook Board.
